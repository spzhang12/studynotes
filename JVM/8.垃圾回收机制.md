## 一、什么是垃圾？

相比于C++，Java增加了内存动态分配和垃圾收集技术，并且摒弃掉了C++中一些比较复杂的概念。

在程序运行中，如果一个对象没有被任何指针所引用，那么这个对象就是需要被回收的垃圾。如果不及时对内存中的垃圾进行清理，那么，这些垃圾对象所占的内存空间回一直保留到应用程序结束，被保留的空间无法被其他对象使用，甚至可能导致内存溢出。

# 二、为什么需要GC？

GC是指回收不再需要使用但是仍占用空间（堆）的对象，释放其占用的空间。除了释放没用的对象，GC也可以清除内存里的记录碎片。碎片整理将所占用的堆内存移到堆的一端，以便JVM将整理出的内存分配给新的对象。如果不进行GC，则内存迟早都会被消耗完，不能保证应用程序的正常进行。而经常造成STW的GC又跟不上实际的需求，所以才会不断地尝试对GC进行优化。

### 垃圾回收的并发与并行

**并行**：**多条垃圾收集线程并行工作**，但此时**用户线程仍处于等待状态**；如ParNew、Parallel Scavenge、Parallel Old；

**串行**：但垃圾收集线程执行；如果内存不够，则程序暂停，启动JVM垃圾回收器进行垃圾回收，回收完再启动程序的线程。

**并发**：**用户线程与垃圾回收线程同时执行（同一时间段内**，也可能会出现STW），垃圾回收线程在执行时不会停顿用户程序的运行；如CMS、G1;

### 安全点

程序执行时并非在所有地方都能够停顿下来开始GC，只有在特定的位置才能停顿下来进行GC，这些位置被称为安全点（safe point）。

为了避免GC等待太长时间，通常会根据“是否**具有让程序长时间执行的特征**”为标准，选择一些执行时间较长的指令作为safe point，如**方法调用、循环跳转和异常跳转**等。

如何保证在GC时，所有线程都跑到最近的安全点停顿下来？

- 抢先式中断（目前未被采用）：先中断所有线程，如果还有线程不在安全点，就恢复线程，让线程跑到安全点；
- 主动式中断：设置一个中断标志，各个线程运行到safe point的时候主动轮询这个标志，如果中断标志为真，则将自己进行中断挂起。

### 安全区域

线程在sleep或者block状态时，不能响应JVM的中断请求，“走”到安全点去中断挂起。

安全区域是指在一段代码片段中，对象的引用关系不会发生变化，在这个区域中的任何位置开始GC都是安全的（扩展的安全点）。

线程运行到安全区域时，标识自己进入了安全区域，当GC时，JVM会忽略位于安全区域的线程；位于安全区域的线程必须等到GC完成才能离开安全区域。

## 引用

在Java中，数据类型分为基本数据类型和引用类型两种，除了8种基本数据类型之外其他的均为引用类型。通过操作引用变量，来对堆中的对象进行操作。

在Java1.2之前，Java并未对引用进行细分，只认为如果一个引用变量存储了另一块内存的起始地址，就将该引用认为是代表了某块内存、某个对象的引用。在这种定义下，一个对象只有被引用和未被引用两种状态。

但是，在一些场景中，存在一种对象，我们希望在内存空间足够时，将其保留在内存中，如果内存在进行垃圾收集后还是非常紧张，则可以抛弃掉这些对象（缓存系统）。

所以，在JDK1.2之后，Java对引用的概念进行了扩充，将引用分为了四类：强引用、软引用、弱引用和虚引用（Phantom）。这四种引用的引用强度依次减弱。引入这四种引用，JVM可以通过垃圾回收器对这四种引用做不同的处理，实现对象生命周期的改变。

Java中默认的给引用变量赋值默认即为强引用。当一个对象被强引用变量引用时，它处于可达状态，是不可能被垃圾回收器回收的。强引用也是造成Java内存泄露的原因之一。

对于一个普通的对象，如果没有其他的引用关系，只要超过了引用的作用域或者显示地将相应的强引用赋值为null，一般认为就是可以被垃圾收集器回收。

```java
Object o1 = new Object();
Object o2 = o1;
o1 = null;
System.gc();  //o1被回收掉，o2强引用o1，一直存在，不会被GC回收
```

**软引用(java.lang.ref.SoftReference)**用来描述一些还有用，但并非必须的对象。这些对象在内存空间充足的时候不会被垃圾收集器回收，但是在系统即将要发生内存溢出之前，会将这些对象进行回收。软引用通常用在对内存敏感的程序中，比如高速缓存就有用到软引用，内存够用的时候保留，不够用就回收（Mybatis中的SoftCache）。

**弱引用(java.lang.ref.WeakReference)**比软引用更弱一些，被弱引用引用的对象在下一次垃圾收集时，不管内存空间够不够用都会被回收。弱引用被用来规范化映射。

虚引用(java.lang.ref.PhantomReference)不会对其引用的对象的生存时间产生任何影响，也无法通过虚引用来访问一个对象。如果一个对象仅被虚引用持有，那么它在任何时候都可能被垃圾收集器回收掉。虚引用不能单独使用，必须和引用队列（ReferenceQueue）联合使用。设置虚引用的唯一目的就是在这个对象被回收器回收的时候收到一个系统通知或者后续添加进一步的处理。

# 垃圾收集器

## 分类

**按垃圾回收的线程数**，垃圾收集器可以分为串行垃圾回收器和并行垃圾回收器。

按工作模式分，可以分为并发式垃圾回收器（用户线程和垃圾回收线程可以同时执行，即在一段时间内）和独占式垃圾回收（STW）器。

按碎片处理方式，可以分为压缩式垃圾回收器和非压缩式垃圾回收器。

- 压缩式垃圾回收器会在回收完成后，对存活对象进行压缩整理，清除回收后的碎片
  - 再分配对象空间使用指针碰撞
- 非压缩式的不会进行碎片整理
  - 再分配对象空间使用空闲链表

按工作的内存区间，可以分为年轻代垃圾回收器和老年代垃圾回收器。

## 评估垃圾回收的性能指标

**吞吐量**：运行用户代码的时间占总运行时间的比例；

**暂停时间**：执行垃圾收集时，程序的工作线程被暂停的时间。

内存占用：Java堆区所占的内存大小

在最大吞吐量优先的情况下，降低停顿时间。
